# Chapter 2: Database Design

## 2.1 What Is MongoDB and Why Is It Used Here?

MongoDB is a **NoSQL database**. Unlike traditional databases (like MySQL) that store data in rigid tables with rows and columns, MongoDB stores data as flexible **documents** inside **collections**.

A **document** looks like a JavaScript object (JSON format):
```json
{
  "_id": "664a1b2c3d4e5f6a7b8c9d0e",
  "fullName": "System Administrator",
  "username": "admin",
  "password": "$2a$10$hashedPasswordHere...",
  "role": "Administrator",
  "status": "Active",
  "createdAt": "2026-05-02T07:30:00.000Z",
  "updatedAt": "2026-05-02T07:30:00.000Z"
}
```

A **collection** is just a group of documents (like a folder of JSON files). This system has **4 collections**:

| Collection | What It Stores |
|-----------|---------------|
| `users` | All user accounts (admins and staff) |
| `categories` | Product groupings (e.g., "Office Supplies", "Cleaning Supplies") |
| `products` | Individual product records with stock counts |
| `stocktransactions` | Every stock-in and stock-out event ever recorded |

## 2.2 How Mongoose Defines the Database Structure

Even though MongoDB is "flexible", we still want rules. Mongoose lets us define a **Schema** — a blueprint that says "every User document MUST have these fields, with these types and constraints."

Each schema is defined in a file inside `backend/models/`.

## 2.3 The User Model

**File:** `backend/models/User.js`

**Purpose:** Stores every person who can log into the system.

### Schema Fields

| Field | Type | Required? | Rules | Purpose |
|-------|------|-----------|-------|---------|
| `fullName` | String | Yes | Trimmed (spaces removed from edges) | The person's real name (e.g., "John Doe") |
| `username` | String | Yes | Unique + Trimmed | The login name (e.g., "jdoe"). No two users can have the same username. |
| `password` | String | Yes | — | The hashed (encrypted) password. The actual password is NEVER stored. |
| `role` | String | No | Must be `"Administrator"` or `"Staff"`. Defaults to `"Staff"`. | Determines what the user can access. |
| `status` | String | No | Must be `"Active"` or `"Inactive"`. Defaults to `"Active"`. | Only Active users can log in. |
| `createdAt` | Date | Auto | Generated by `timestamps: true` | When the account was created. |
| `updatedAt` | Date | Auto | Generated by `timestamps: true` | When the account was last modified. |

### What Happens Behind the Scenes
- When a new user is created, the controller hashes the password using `bcryptjs` with a salt round of 10 before saving it.
- The `unique: true` on `username` means MongoDB will reject any attempt to create a second user with the same username.
- The `enum` on `role` and `status` means MongoDB will reject any value that is not in the allowed list.

### Example Document in MongoDB
```json
{
  "_id": "664a1b2c3d4e5f6a7b8c9d0e",
  "fullName": "System Administrator",
  "username": "admin",
  "password": "$2a$10$K6VfE8x...(hashed)...",
  "role": "Administrator",
  "status": "Active",
  "createdAt": "2026-05-02T07:30:00.000Z",
  "updatedAt": "2026-05-18T06:35:00.000Z"
}
```

## 2.4 The Category Model

**File:** `backend/models/Category.js`

**Purpose:** Groups products into logical categories for easier organization.

### Schema Fields

| Field | Type | Required? | Rules | Purpose |
|-------|------|-----------|-------|---------|
| `categoryName` | String | Yes | Unique + Trimmed | The category label (e.g., "Office Supplies"). Must be unique. |
| `description` | String | No | Defaults to `""` | An optional explanation of what belongs in this category. |
| `status` | String | No | `"Active"` or `"Inactive"`. Defaults to `"Active"`. | Allows soft-deletion (hiding without deleting). |
| `createdAt` | Date | Auto | — | When the category was created. |
| `updatedAt` | Date | Auto | — | When the category was last modified. |

### Why Categories Exist
Instead of every product being a flat list, categories let you organize:
- "Office Supplies" → Bond Paper, Ballpen, Stapler
- "Cleaning Supplies" → Alcohol, Mop, Detergent

This makes filtering and reporting much easier.

## 2.5 The Product Model

**File:** `backend/models/Product.js`

**Purpose:** Stores every inventory item the organization tracks.

### Schema Fields

| Field | Type | Required? | Rules | Purpose |
|-------|------|-----------|-------|---------|
| `productCode` | String | Yes | Unique + Trimmed | A short identifier (e.g., "P-001"). Must be unique across all products. |
| `productName` | String | Yes | Trimmed | The descriptive name (e.g., "Bond Paper"). |
| `categoryId` | ObjectId | Yes | References the `Category` collection | Links this product to a category. This is a **foreign key relationship**. |
| `description` | String | No | Defaults to `""` | Optional details about the product. |
| `unitOfMeasure` | String | Yes | — | How the product is counted (e.g., "Ream", "Bottle", "Box", "Kg"). |
| `quantityInStock` | Number | No | Minimum: 0, Defaults to 0 | The current count of this product in the warehouse. |
| `reorderLevel` | Number | No | Minimum: 0, Defaults to 0 | The threshold. If `quantityInStock ≤ reorderLevel`, the product is flagged as "Low Stock". |
| `price` | Number | No | Minimum: 0, Defaults to 0 | The unit price of the product. |
| `status` | String | No | `"Active"` or `"Inactive"`. Defaults to `"Active"`. | Soft-deletion flag. |
| `createdAt` | Date | Auto | — | — |
| `updatedAt` | Date | Auto | — | — |

### The Category Relationship (Foreign Key)
The `categoryId` field does not store the category name directly. Instead, it stores the **MongoDB ObjectId** (a unique identifier) of the category document. This is called a **reference**.

When the backend needs to show the category name alongside a product, it uses Mongoose's `.populate("categoryId", "categoryName")` method, which automatically looks up the referenced category document and replaces the ID with the actual category data.

### Example Document in MongoDB
```json
{
  "_id": "664b2c3d4e5f6a7b8c9d0e1f",
  "productCode": "P-001",
  "productName": "Bond Paper",
  "categoryId": "664a1b2c3d4e5f6a7b8c9d0e",
  "description": "A4 bond paper",
  "unitOfMeasure": "Ream",
  "quantityInStock": 10,
  "reorderLevel": 5,
  "price": 250,
  "status": "Active",
  "createdAt": "2026-05-02T08:00:00.000Z",
  "updatedAt": "2026-05-18T06:35:00.000Z"
}
```

## 2.6 The StockTransaction Model

**File:** `backend/models/StockTransaction.js`

**Purpose:** Records every single stock movement (incoming or outgoing) as a permanent, immutable audit log.

### Schema Fields

| Field | Type | Required? | Rules | Purpose |
|-------|------|-----------|-------|---------|
| `productId` | ObjectId | Yes | References `Product` | Which product this transaction is about. |
| `transactionType` | String | Yes | Must be `"Stock In"` or `"Stock Out"` | Whether inventory was added or removed. |
| `quantity` | Number | Yes | Minimum: 1 | How many units were moved. |
| `previousQuantity` | Number | Yes | — | The product's stock count BEFORE the transaction. |
| `newQuantity` | Number | Yes | — | The product's stock count AFTER the transaction. |
| `processedBy` | ObjectId | Yes | References `User` | Which user performed this transaction. |
| `remarks` | String | No | Defaults to `""` | An optional note (e.g., "Monthly restock", "Damaged items returned"). |
| `createdAt` | Date | Auto | — | The exact timestamp of the transaction. |

### Why `previousQuantity` and `newQuantity` Are Stored
These fields create an **audit trail**. Even if someone changes the product's current stock count later, you can always trace back through the transaction history to see exactly how inventory changed over time.

**Example of a Stock-In transaction:**
```
previousQuantity: 10  →  quantity: 5 added  →  newQuantity: 15
```

**Example of a Stock-Out transaction:**
```
previousQuantity: 15  →  quantity: 3 removed  →  newQuantity: 12
```

## 2.7 Entity Relationship Diagram

This diagram shows how the four collections relate to each other:

```
┌──────────────────┐         ┌──────────────────┐
│      User        │         │     Category     │
├──────────────────┤         ├──────────────────┤
│ _id              │         │ _id              │
│ fullName         │         │ categoryName     │
│ username (unique)│         │ description      │
│ password (hashed)│         │ status           │
│ role             │         │ createdAt        │
│ status           │         │ updatedAt        │
│ createdAt        │         └────────┬─────────┘
│ updatedAt        │                  │
└────────┬─────────┘                  │ one-to-many
         │                            │ (one category has many products)
         │                            ▼
         │                   ┌──────────────────┐
         │                   │     Product      │
         │                   ├──────────────────┤
         │                   │ _id              │
         │ one-to-many       │ productCode      │
         │ (one user         │ productName      │
         │  processes many   │ categoryId ──────┘ (FK → Category)
         │  transactions)    │ description      │
         │                   │ unitOfMeasure    │
         │                   │ quantityInStock  │
         │                   │ reorderLevel     │
         │                   │ price            │
         │                   │ status           │
         │                   └────────┬─────────┘
         │                            │
         │                            │ one-to-many
         │                            │ (one product has many transactions)
         │                            ▼
         │                   ┌──────────────────────┐
         │                   │  StockTransaction    │
         │                   ├──────────────────────┤
         └──────────────────►│ _id                  │
                             │ productId ───────────┘ (FK → Product)
                             │ transactionType      │
                             │ quantity             │
                             │ previousQuantity     │
                             │ newQuantity          │
                             │ processedBy ─────────┘ (FK → User)
                             │ remarks              │
                             │ createdAt            │
                             └──────────────────────┘
```

### Relationships Summarized
| From | To | Type | Meaning |
|------|----|------|---------|
| Category → Product | `categoryId` | One-to-Many | One category can contain many products |
| Product → StockTransaction | `productId` | One-to-Many | One product can have many stock transactions |
| User → StockTransaction | `processedBy` | One-to-Many | One user can process many stock transactions |

## 2.8 Database Connection

**File:** `backend/config/db.js`

This file contains a single function called `connectDB()` that:
1. Reads the `MONGO_URI` value from the `.env` file
2. Uses Mongoose to connect to MongoDB Atlas
3. Logs "MongoDB connected successfully" if it works
4. Terminates the entire application (`process.exit(1)`) if the connection fails

This function is called once when the server starts in `app.js`.
